STEP C — Build Make.com scenario:

1. Open Make.com > Create a new scenario

2. Add Trigger module:
   - Search for: HubSpot CRM
   - Select: Watch CRM Objects
   - Object type: Contacts
   - Configure the connection with your HubSpot Private App token (HUBSPOT_API_KEY)
   - Field to watch: smykm_trigger_ts (the property you created in Step A)
   - Polling interval: 15 minutes (or minimum available on your Make.com plan)

3. Add a Filter (optional but recommended):
   - Only continue if smykm_trigger_ts is not empty

4. Add HTTP module:
   - Search for: HTTP
   - Select: Make a request
   - Configure:
     - URL: https://api.github.com/repos/staffdomain-devops/SMYKM/actions/workflows/campaign.yml/dispatches
     - Method: POST
     - Headers (add each separately):
       * Authorization: Bearer {paste the smykm-make-dispatch PAT value}
       * Accept: application/vnd.github+json
       * X-GitHub-Api-Version: 2022-11-28
     - Body type: Raw
     - Content type: application/json
     - Request content (exact JSON):
       {
         "ref": "main",
         "inputs": {
           "contact_id": "{{toString(1.id)}}",
           "contact_email": "{{1.email}}"
         }
       }
       NOTE: Replace 1.id and 1.email with the actual Make.com variable paths from your Watch CRM Objects output. The key requirement is contact_id MUST be wrapped in toString() to coerce it from integer to string.

   - Set to: OFF (disabled)
   - This ensures HTTP 204 from GitHub is treated as success, not error.
   - If you cannot find this setting, add an error handler that ignores 204 responses.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Make.com → GitHub API | External service sends authenticated POST; PAT is the credential |
| HubSpot Workflow → Make.com trigger property | HubSpot sets smykm_trigger_ts; Make.com reads it — trust boundary is HubSpot's data integrity |
| Make.com HTTP module → GitHub dispatch inputs | contact_id and contact_email injected from HubSpot data; treated as untrusted strings by the workflow |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-07 | Elevation of Privilege | smykm-make-dispatch PAT scope creep | mitigate | PAT scoped to Actions: write only. No Contents: write. Scoped to single repo (staffdomain-devops/SMYKM). Rotate annually. |
| T-00-08 | Information Disclosure | PAT value stored in Make.com | accept | Make.com encrypts stored credentials. PAT has minimal scope (Actions: write only) so exposure risk is bounded. Calendar reminder to rotate before expiry. |
| T-00-09 | Spoofing | Rogue actor triggers workflow_dispatch via stolen PAT | mitigate | PAT can only trigger workflow_dispatch (not write code, not push commits). Worst case: a spurious run is queued, fails cleanly (no contact data exists), and appears in the audit log. Rate-limiting mitigated by PAT rotation on detection. |
| T-00-10 | Tampering | Make.com injects non-string contact_id causing 422 | mitigate | Make.com toString() coercion applied to contact_id before injection into JSON body. Acceptance criterion requires 204 (not 422) on test run. |
| T-00-11 | Denial of Service | Make.com fires duplicate dispatches for same contact | mitigate | Per-contact concurrency group (established in Plan 01) queues duplicates. No run is lost or race-conditioned. |
| T-00-12 | Information Disclosure | PAT stored as plaintext in Make.com HTTP module URL or body | mitigate | PAT stored in Make.com credential/connection store, referenced as a variable — never hardcoded in URL or body JSON visible in scenario export. |
</threat_model>
